# 真实音频处理接入评审

本文件用于阶段 3：在真正接入 AI 转写前，先把技术选择、费用风险、隐私风险和最小接入范围讲清楚。

## 当前结论

阶段 3 先做接入前评审，并预留接口，不直接调用付费服务。

用户已确认：先不接入 AI，等产品功能整体完成后，最后再接入真实 AI。

推荐第一步采用：

- 服务方向：OpenAI Audio Transcriptions API
- 推荐模型：`gpt-4o-mini-transcribe`
- 备选模型：`gpt-4o-transcribe`
- 说话人区分备选：`gpt-4o-transcribe-diarize`
- 保守兼容备选：`whisper-1`

大白话解释：

- `gpt-4o-mini-transcribe`：适合作为第一版真实转写入口，成本和能力比较适合试运行。
- `gpt-4o-transcribe`：更偏质量优先，适合后面提高准确率。
- `gpt-4o-transcribe-diarize`：可以做说话人区分，但更适合后面单独评估。
- `whisper-1`：老牌转写模型，兼容一些字幕格式输出，但不是第一推荐。

## 官方资料摘录后的项目判断

根据 OpenAI 官方文档，语音转文字使用 Audio API 的 `audio/transcriptions` 接口。

官方文档提到的可用转写模型包括：

- `gpt-4o-mini-transcribe`
- `gpt-4o-transcribe`
- `gpt-4o-transcribe-diarize`
- `whisper-1`

官方文档还说明，上传文件当前有大小限制，且常见支持格式包括：

- `mp3`
- `mp4`
- `mpeg`
- `mpga`
- `m4a`
- `wav`
- `webm`

参考链接：

- OpenAI Speech to text 文档：https://platform.openai.com/docs/guides/speech-to-text
- OpenAI Audio API 参考：https://platform.openai.com/docs/api-reference/audio/createTranscription
- OpenAI gpt-4o-transcribe-diarize 模型说明：https://developers.openai.com/api/docs/models/gpt-4o-transcribe-diarize

## 第一版真实接入建议

当前先不执行真实接入，只保留接口。

已预留：

- `GET /api/transcriptions/audio`
- `POST /api/transcriptions/audio`

详细说明见：

- `docs/接口预留说明.md`

等项目整体功能完成后，再执行下面的真实接入建议。

第一版真实接入只做一件事：

用户上传一个音频文件，系统返回一段转写文字。

不建议第一步同时做：

- 视频转音频。
- 大文件切片。
- 说话人区分。
- 自动会议纪要。
- 字幕时间轴。
- 团队权限。
- 付费套餐。

原因很简单：先确认“能不能稳定把声音变成文字”，再继续加复杂能力。

## 文件处理建议

第一版建议采用临时处理：

1. 用户在网页选择音频文件。
2. 后端接口收到文件。
3. 后端把文件转交给转写服务。
4. 后端拿到文字结果。
5. 后端把文字结果返回给页面。
6. 后端不长期保存原始文件。

大白话解释：

用户文件只用于本次处理，不先做长期文件库。这样隐私风险更低，开发也更简单。

## 隐私和风险

真实接入前必须确认：

- 用户是否允许文件发送到第三方 AI 服务。
- 文件是否需要保存。
- 文件保存多久。
- 用户如何删除文件。
- 是否需要在页面明确提示“文件会被发送到 AI 服务处理”。
- 是否需要限制文件大小。
- 是否需要限制文件类型。

## 费用风险

真实 AI 服务通常按使用量收费。

大白话解释：

不是网站打开就一定花钱，而是用户上传音频、调用 AI 转写时可能产生费用。

所以真实接入前要做：

- 限制单个文件大小。
- 限制单次处理时长。
- 给用户显示“当前是测试功能”。
- 不要在没有确认前开放给大量用户。

## 最小验收标准

阶段 3 如果继续到真实接口实现，最小验收标准是：

- 没有 API Key 时，页面给出清楚提示，不报奇怪错误。
- 只接受常见音频格式。
- 文件过大时给出友好提示。
- 成功后能在页面看到转写文字。
- 失败时能显示失败原因。
- 不保存用户原始音频文件。
- 不做自动扣费以外的隐藏操作。
