# 技术方案

## 已选择的技术

项目第一阶段使用：

- Next.js
- React
- TypeScript

大白话解释：

- Next.js：用来搭建网站的工具，适合以后做登录、页面跳转、后台接口。
- React：用来做页面按钮、卡片、表格、编辑器这些界面部件。
- TypeScript：让代码更容易检查错误，减少写错东西的概率。

## 第一阶段技术策略

第一阶段只做可点击原型。

也就是说：

- 页面可以打开。
- 按钮可以点击。
- 示例数据可以显示。
- 一些文字可以编辑。
- 但不会真的上传大文件到 AI 服务。
- 不会真的从音频或视频里识别文字。

第一阶段会使用假数据。

假数据的意思是：提前写好一些示例会议、示例字幕、示例任务状态，让页面看起来像真实运行。这样可以先验证产品流程。

## 为什么先不用真实 AI

真实音视频处理比较复杂，通常需要处理：

- 文件上传。
- 文件大小限制。
- 语音识别服务。
- 说话人区分服务。
- 视频转音频。
- 处理时间较长的任务队列。
- 服务费用。
- 用户隐私和文件删除。

如果一开始全部做，项目很容易变慢，也很难判断是页面问题还是 AI 服务问题。

所以项目先做页面和流程，再接真实能力。

## 后续真实能力方向

后续进入阶段 3 前，需要单独确认这些选择：

1. 语音转文字服务
   - 可以考虑 OpenAI Whisper 或其他语音识别服务。
   - 作用：把录音或视频里的声音变成文字。

2. 说话人区分服务
   - 作用：判断哪句话是谁说的。
   - 这类能力通常需要专门服务，不一定和语音转文字是同一个服务。

3. 视频处理工具
   - 可以考虑 FFmpeg。
   - 作用：从视频里取出声音，方便生成字幕。

4. 文件存储
   - 需要决定文件放在哪里。
   - 早期可以用本地存储。
   - 正式上线通常会用对象存储服务。

5. 任务处理
   - 音视频处理可能要等几十秒或几分钟。
   - 后续可能需要任务队列。
   - 大白话：任务队列就像排队叫号，文件一个个处理，不让网站卡住。

## 阶段 3 接入前评审结论

阶段 3 先做真实接入前评审，并预留接口，不直接调用付费服务。

用户已确认：先不接入 AI，等网站功能整体完成后，最后再接入真实 AI。

推荐第一步：

- 优先评估 OpenAI Audio Transcriptions API。
- 第一版真实接入只做音频转文字。
- 推荐先评估 `gpt-4o-mini-transcribe`。
- `gpt-4o-transcribe` 作为质量优先备选。
- `gpt-4o-transcribe-diarize` 作为说话人区分备选。
- `whisper-1` 作为兼容备选。

第一版真实接入暂时不做：

- 视频转音频。
- 大文件切片。
- 说话人区分。
- 自动会议纪要。
- 字幕时间轴。
- 付费系统。

详细评审见：

- `docs/真实音频处理接入评审.md`
- `docs/接口预留说明.md`

## 代码质量要求

- 优先写简单清楚的代码。
- 不为了炫技加复杂工具。
- 页面结构要方便以后接真实后台。
- 专业命名可以保留，但文档里必须解释清楚。
- 每个阶段完成后更新开发日志。
